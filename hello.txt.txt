🧠 What autoresearch actually is (strip the hype)

At its core, it’s not “AI doing research.”

It’s a tight optimisation loop wrapped in an agent:

propose change → run experiment → measure → accept/reject → repeat

That’s it.

The key innovation is discipline + constraints, not intelligence.

The canonical loop
Agent edits code (train.py)
Runs a time-boxed experiment (~5 min)
Evaluates using one fixed metric
Keeps or discards change
Repeats ~100x overnight
⚙️ The architecture (this is what you actually need)

Autoresearch is basically 3 layers:

1. Control layer (YOU define this)
program.md = strategy / objective
Defines:
what success means
constraints
allowed actions

👉 This is the real product surface

2. Agent layer
LLM (Claude / GPT / local)
Reads logs + instructions
Proposes next change
3. Execution harness
Runs experiments
Returns metric
Handles rollback / logging
4. Search loop (the core engine)
while True:
    proposal = agent.generate_change(context)

    apply_change(proposal)

    result = run_experiment()

    if result > best:
        keep_change()
        best = result
    else:
        revert()

    log_everything()
🚫 What people misunderstand

Let’s be blunt:

❌ It’s NOT self-improving intelligence
❌ It doesn’t invent new goals
❌ It’s not autonomous AGI

It’s closer to:

Automated hill climbing with an LLM as the mutation generator

The power comes from:

fast iteration
strict evaluation
persistence
🔥 How to integrate into YOUR project (this is the important part)

You said you don’t care if it’s ML training — good.
Because the pattern is universal.

🧩 Step 1: Define your “search surface”

Ask:

“What can the agent safely modify?”

Examples:

If you're building a trading system:
strategy parameters
signal logic
portfolio weights
If you're building a game AI:
decision rules
reward functions
heuristics
If you're building SaaS:
pricing logic
prompts
ranking algorithms

👉 Limit this HARD (like autoresearch only edits one file)

📏 Step 2: Define ONE metric

This is critical.

Examples:

trading → Sharpe ratio / PnL
model → validation loss
product → conversion rate

If your metric is weak → your system is useless

⏱️ Step 3: Time-box experiments

Autoresearch forces:

fixed runtime
comparable results

You should do the same:

MAX_RUNTIME = 300  # seconds

This prevents:

overfitting
cheating
inconsistent results
🤖 Step 4: Plug in an agent (LLM)

Your agent prompt should look like:

You are optimizing X.

Goal: maximize <metric>

Constraints:
- Only modify <allowed area>
- Keep changes minimal
- Avoid breaking system

Previous results:
<logs>

Propose next change.
📊 Step 5: Build a feedback loop

Log EVERYTHING:

change diff
metric result
decision (keep/reject)

This becomes:

training data
memory
future improvement signal
🧠 Step 6: Add memory (this is where you go beyond autoresearch)

Karpathy’s version is stateless-ish.

You can level it up:

store best strategies
cluster successful patterns
avoid repeating failures

👉 This turns it from:

brute-force search → learning system

🚀 Advanced: Turn this into a real system (your edge)

Here’s where you outperform the original repo:

1. Multi-agent competition
multiple agents propose different ideas
best one wins
2. Parallel experiments
run 10–100 at once
massively increases search speed
3. Meta-optimization
agent improves its OWN prompting strategy
4. Cross-domain transfer
reuse winning patterns across tasks
